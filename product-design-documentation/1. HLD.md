# 🏷️ **ChoreLoop: Flat-Trust Task Ownership with Soft Accountability**

### High-Level Design (HLD)

---

## 1. Product Overview

**ChoreLoop** is a streamlined, collaborative task-management application designed for a single household. It operates on a **Flat Trust Model**, where all users have equal permissions, combined with **Soft Accountability Mechanisms** to ensure transparency and responsibility without introducing hierarchy.

The system emphasizes:

* frictionless interaction
* clear ownership
* visible accountability
* minimal administrative overhead

### **Value Proposition**

Effortless chore coordination through a **Claim → Own → Complete loop**, reinforced by transparency instead of control.

### **Target Users**

Members of a shared household (adults, children, or mixed).

### **Core Philosophy**

* No roles, no permissions, no barriers
* Accountability through visibility, not restriction
* Designed for real-world human behavior, not idealized cooperation

## 4.1 Chore Lifecycle

### **States**

* Available
* Claimed
* Completed
* Overdue
* Archived

---

### **Create**

* Any user creates chore:

  * Title
  * Description
  * Due date
* Automatically sets:

  * `created_by`

---

### **Claim**

* “I’ll do it!” → assigns chore to user
* Sets:

  * `assigned_to`
  * `claimed_at`

---

### **Take Over (Reassignment)**

* Available when chore is owned by another user

**Flow:**

1. Tap “Take Over”
2. Confirmation dialog:

   > “This chore is assigned to [X]. Take over?”
3. Confirm → reassigned

---

### **Unclaim**

* Available only to current owner
* Returns chore to Available state

---

### **Complete**

* Only visible to assigned user

**Flow:**

1. Tap “Done!”
2. Confirm
3. Sets:

   * `status = completed`
   * `completed_by`
   * `completed_at`

---

### **Archive**

* Any user can archive a chore

**Flow:**

1. Tap “Archive”
2. Confirm
3. Sets:

   * `archived_at`
   * `archived_by`

**Post-action:**

* Snackbar with Undo

## 5. Web App Architecture (Modern Vanilla)

### **Pattern**
* **Component-Based State Management**: A reactive "State-to-UI" approach using a global state object and render functions.
* **Modular JS**: ES6 Modules for separation of concerns (API, State, UI Components).

### **UI & Styling**
* **CSS3 Custom Properties**: For theming and consistent spacing.
* **Flexbox/Grid**: For the responsive dashboard layout.
* **Mobile-First Design**: Optimized for kitchen tablets and smartphones.

### **State & Identity**
* **localStorage**: To persist `currentUser` identity and session data.
* **URL Hash Routing**: Simple single-page application (SPA) navigation.

### **Data Strategy**
* **Fetch API**: For asynchronous communication with the PHP backend.
* **Polling/Refetch**: Simple interval-based data refreshing to maintain SSOT (Source of Truth) across different household devices.

---

## 5.1 Web Implementation Details

### **Core Components**

#### **1. The State Store**
A central object that triggers re-renders whenever data changes.
```javascript
const Store = {
  user: JSON.parse(localStorage.getItem('chore_user')),
  chores: [],
  logs: [],
  listeners: [],
  
  setState(key, value) {
    this[key] = value;
    this.listeners.forEach(fn => fn(this));
  }
};
```

#### **2. Responsive Grid System**
The dashboard utilizes a CSS Grid that adapts from a single column (mobile) to a four-column layout (desktop/tablet).
```css
.dashboard-grid {
  display: grid;
  gap: 1.5rem;
  grid-template-columns: repeat(auto-fit, minmax(300px, 1fr));
}
```

#### **3. Component Logic**
* **ChoreCard**: A reusable template literal function that renders specific action buttons based on the chore state and current user identity.
* **IdentityPicker**: An overlay that blocks app access until a user profile is selected.

## 5.3 Web-Specific UX Patterns

### **Optimistic UI Updates**
To maintain the "frictionless" promise, the Web UI updates the local state immediately upon "Claim" or "Complete," reverting only if the server returns a conflict error.

### **Action Attribution Tooltips**
Hovering (on desktop) or tapping (on mobile) the "Claimed by" badge reveals the timestamp and activity log snippet for that specific chore.

### **Visual Cues**
* **Pulse Animation**: For chores that are overdue.
* **Sticky Header**: Keeps the "Acting as: [User]" visible while scrolling through long chore lists.
