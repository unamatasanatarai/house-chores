# High-Level Design (HLD): Family Chores App (Flat Trust Model)

## 1. Product Overview
The **Chores App** is a streamlined, collaborative task-management platform designed for a single-family unit. It operates on a **"Flat Trust Model,"** where every family member has identical permissions. The app focuses on zero-friction coordination by removing passwords and hierarchy, replacing them with shared accountability and a clear ownership loop.

* **Value Proposition:** Frictionless chore coordination through a "Locked Ownership Model" that fosters accountability without administrative bottlenecks.
* **Target Users:** All members of a single household.
* **Core Philosophy:** Open access, shared responsibility, and physical-proximity-based security.

---

## 2. Scope

### **In-Scope (MVP)**
* **Family & User Setup:** One-time family creation and simple user name entry.
* **Persistent Identity:** Mandatory user selection remembered across app restarts.
* **Universal Management:** Any user can create, view, claim, and delete any chore.
* **Safety Guards:** A confirmation dialog for all deletion actions.
* **Live Identity Tracking:** Active user’s name persistently visible in the top navigation bar.
* **Claiming System:** "Locked Ownership" where a chore is tied to the claimant until completion.
* **Online Requirement:** All actions require an active internet connection. The Server Database is the Single Source of Truth (SSOT).

### **Out-of-Scope**
* User authentication (Emails, Passwords, Social Login).
* Avatars or media uploads.
* Role-based permissions (Parent vs. Child).
* Gamification or points systems.

---

## 3. Users & Identity
### **The Shared Experience**
* **No Roles:** There is no "Admin" or "Child" distinction. Every user has a "User" role with full CRUD (Create, Read, Update, Delete) permissions.
* **Identity Selection:** On the first launch (or when switching), the app presents a list of names. Once a user is selected, the app persists this choice in local storage.
* **Persistent Visibility:** To prevent accidental actions on someone else's behalf, the active user's name is always displayed in the `TopAppBar`.

---

## 4. Functional Requirements

### **4.1 Onboarding & Setup**
* **Family Creation:** The first user creates a Family Name.
* **User Entry:** Users are added via text input (Name only). 
* **Identity Lock:** The app requires a user to "Pick who you are." This selection is saved in the device's `DataStore`.

### **4.2 Chore Lifecycle**
* **Creation:** Any user can create a chore (Title, Description, Due Date).
* **Claiming:** Any user can tap **"I’ll do it!"** on an available chore. The chore moves to the "Claimed" state and is assigned to that user.
* **Deletion:** Any user can delete a chore. Upon tapping delete, a **"Confirm Deletion"** dialog appears to prevent accidental loss of data.
* **Completion:** The assigned user marks the chore as "Done!" to trigger completion animations and status updates.

### **4.3 UI/UX Elements**
* **Navigation Bar:** A persistent top bar displaying "Acting as: [Name]" and a "Switch User" action.
* **Dialogs:** Standard Material Design alerts for destructive actions.

---

## 5. Android App Architecture
* **Pattern:** MVVM (Model-View-ViewModel) with Clean Architecture.
* **UI Layer:** Jetpack Compose for a modern, state-driven interface.
* **Identity Management:** A `UserSessionManager` using **Jetpack DataStore** to persist the `activeUserId`.
* **Local Data:** Room Database is not strictly required as an SSOT since the app requires an active connection, but can be used for caching.
* **Network Layer:** Standard asynchronous API calls handle communication with the PHP API.

---

## 6. Backend & Integration

### **Technology Stack**
* **Language:** PHP.
* **Database:** MySQL.
* **Interface:** RESTful API.

### **Authentication & Security**
* **Simple Auth:** The API uses a **Hardcoded Token** for authentication.
* **Implementation:** Every request from the Android app must include an `X-API-TOKEN` or `Authorization: Bearer [TOKEN]` header. The PHP backend validates this token against a constant value defined in the server configuration.

### **API Endpoints (High-Level)**
* `POST /api/family/create`: Initialize a family.
* `POST /api/users/add`: Add a new user to the family.
* `GET /api/chores`: Fetch all family chores.
* `POST /api/chores/sync`: Batch sync local changes (create, claim, delete, complete).

---

## 7. Data Management & Connectivity Strategy
### **Always-Online Logic**
1.  **Server as SSOT:** The application assumes users are always online. If offline, the app does not allow changes.
2.  **Immediate API Call:** When a user claims a chore, the app makes an immediate API call.
3.  **Conflict Resolution:** "First to the Server Wins." The PHP backend processes the first request it receives. Subsequent claims on the same chore are rejected.

---

## 8. Key User Flows

### **8.1 User Identity Flow**
1.  User opens the app.
2.  System checks DataStore for `activeUserId`.
3.  If empty: Show "Who are you?" list.
4.  If present: Load Dashboard with the user's name in the **Top Navigation Bar**.

### **8.2 Safe Deletion Flow**
1.  User taps the "Delete" icon on a chore card.
2.  **System Action:** Interrupts the flow with a "Confirm Deletion" dialog.
3.  User taps "Delete" again.
4.  Chore is soft-deleted on the server via the PHP API (`deleted_at = NOW()`), and removed from the active lists.

---

## 9. Technical Data Model (MySQL)

### **Users Table**
* `id` (VARCHAR(36), PK, UUID)
* `family_id` (VARCHAR(36))
* `name` (VARCHAR)
* `updated_at` (TIMESTAMP)

### **Chores Table**
* `id` (VARCHAR(36), PK, UUID)
* `family_id` (VARCHAR(36))
* `title` (VARCHAR)
* `description` (TEXT)
* `status` (VARCHAR(20): Available, Claimed, Completed, Overdue)
* `assigned_to` (VARCHAR(36), FK to Users)
* `due_date` (DATETIME)
* `deleted_at` (DATETIME) - Used for soft deletes.
* `updated_at` (TIMESTAMP)

---

## 10. Assumptions & Risks
* **Assumptions:**
    * The "Hardcoded Token" provides sufficient security for a private family app.
    * Physical access to the device is the primary security layer.
* **Risks:**
    * **Accidental Deletion:** Even with a confirmation dialog, the flat trust model allows anyone to delete any chore.
    * **Token Exposure:** A hardcoded token in the APK can be extracted; this risk is accepted for the MVP simplicity.

---

## 11. MVP Release Plan
* **Phase 1:** PHP/MySQL API setup with hardcoded token validation.
* **Phase 2:** Android Onboarding and Name-based identity persistence.
* **Phase 3:** Core Chore Dashboard with Top Bar identity display.
* **Phase 4:** Deletion dialogs and soft-delete API integration.
