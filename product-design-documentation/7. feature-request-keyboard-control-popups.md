# **Feature Request Document: Web Keyboard Interaction**

**Feature Title:** Keyboard Interaction & Shortcut Affordance (Escape/Enter)  
**Status:** Complete  
**Target:** Web (Vanilla JS)  

---

## **1. Problem Statement**
Currently, users must interact with the UI via pointer/mouse for every action. For high-frequency tasks like closing "Take Over" dialogs, or confirming a chore creation, this introduces unnecessary friction. We need to standardize "Safety" and "Confirmation" keys to improve user velocity on the web platform.

---

## **2. Functional Requirements**

### **FR.1: Global Escape Key (Dismissal)**
* **Behavior:** Pressing the `Esc` key must immediately close the currently active top-level UI element.
* **Priority Order:** 1.  Confirmation Dialogs / Modals.
    2.  The Identity Selection overlay (if applicable/cancellable).
* **Non-Action:** If no modal or dismissible overlay is open, the `Esc` key should perform no action (should not affect browser-level state).

### **FR.2: Enter Key (Confirmation)**
* **Behavior:** Pressing the `Enter` key (or `Return`) while a confirmation dialog is active or a form is focused must trigger the "Positive" action (e.g., "Confirm Take Over" or "Submit New Chore").
* **Safety Check:** `Enter` should only trigger the action if the "Positive" button is currently **enabled** (e.g., if a title is required and the field is empty, `Enter` does nothing).

### **FR.3: Visual Shortcut Affordance (Icons)**
* **Requirement:** Buttons within dialogs and forms must display a visual hint of their associated shortcut key.
* **Visual Design:** * **Confirm/Submit buttons:** Append a small `[↵]` icon.
    * **Cancel/Close buttons:** Append a small `[Esc]` icon.
* **Styling:** Use a low-contrast, mono-spaced font and a "key-cap" style CSS border (e.g., light gray background, subtle rounding) to distinguish the shortcut from the button label.

---

## **3. Technical Implementation Tasks (Web)**

* **W-FR.1:** Add a global `keydown` listener to the `window` object in `app.js` to intercept key events.
* **W-FR.2:** Implement a **Modal Stack** manager (array-based) to track open overlays, ensuring the `Esc` key only closes the most recently opened (top-most) element.
* **W-FR.3:** Update the `ChoreCard` and `Dialog` JavaScript templates to include inline SVG icons for Enter and Escape keys.
* **W-FR.4:** Ensure `event.preventDefault()` is used on the `Enter` key within forms to prevent unwanted page refreshes while still triggering the specific API logic.

---

## **4. UX & Acceptance Criteria**

1.  **Scenario: Dismissing a Dialog**
    * **Action:** User opens the "Take Over" dialog and presses `Esc`.
    * **Result:** Dialog closes immediately; focus returns to the previous element; no data is sent to `/api/chores/{id}/claim`.
2.  **Scenario: Fast Chore Creation**
    * **Action:** User is typing a new chore title in the creation modal and presses `Enter`.
    * **Result:** The `POST /api/chores/add` request is triggered immediately if validation passes.
3.  **Scenario: Visual Clarity**
    * **Observation:** When any modal appears, the "Confirm" button clearly displays a visible Enter key icon `[↵]`, and the "Cancel" button displays an `[Esc]` icon.
