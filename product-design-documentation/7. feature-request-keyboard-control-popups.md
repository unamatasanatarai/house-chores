# **Part 2: Feature Request Document**

**Feature Title:** Keyboard Interaction & Shortcut Affordance (Escape/Enter)  
**Status:** Draft  
**Target:** Web (Vanilla JS) & Android (Jetpack Compose)  

---

## **1. Problem Statement**
Currently, users must interact with the UI via pointer/touch for every action. For high-frequency tasks like dismissing snackbars, closing "Take Over" dialogs, or confirming a chore creation, this introduces unnecessary friction. We need to standardize "Safety" and "Confirmation" keys to improve velocity.

---

## **2. Functional Requirements**

### **FR.1: Global Escape Key (Dismissal)**
* **Behavior:** Pressing the `Esc` key must immediately close the currently active top-level UI element.
* **Priority Order:** 1.  Confirmation Dialogs / Modals.
    2.  The Identity Selection overlay (if applicable/cancellable).
    3.  Active Snackbars (dismiss early).
* **Non-Action:** If no modal is open, the `Esc` key should perform no action (should not exit the app).

### **FR.2: Enter Key (Confirmation)**
* **Behavior:** Pressing the `Enter` key (or `Return`) while a confirmation dialog is active must trigger the "Positive" action (e.g., "Confirm Take Over" or "Submit New Chore").
* **Safety Check:** `Enter` should only work if the "Positive" button is enabled (e.g., if a title is required and the field is empty, `Enter` does nothing).

### **FR.3: Visual Shortcut Affordance (Icons)**
* **Requirement:** Buttons within dialogs and forms must display a visual hint of their associated key.
* **Visual Design:** * **Confirm/Submit buttons:** Append a small `[↵]` icon.
    * **Cancel/Close buttons:** Append a small `[Esc]` icon.
* **Styling:** Use a low-contrast, mono-spaced font or a "key-cap" style CSS/Compose border to distinguish the shortcut from the button text.

---

## **3. Technical Implementation Tasks**

### **Web (Vanilla JS)**
* **W-FR.1:** Add a global `keydown` listener to the `window` object in `app.js`.
* **W-FR.2:** Implement a "Modal Stack" manager to ensure `Esc` only closes the top-most active element.
* **W-FR.3:** Update the `ChoreCard` and `Dialog` templates to include SVG icons for Enter/Escape keys.

### **Android (Jetpack Compose)**
* **A-FR.1:** Use `Modifier.onKeyEvent` on the root `Scaffold` or individual `Dialog` composables to intercept hardware key events.
* **A-FR.2:** Add `KeyboardOptions` to text fields to ensure `ImeAction.Done` maps correctly to the `Enter` confirmation logic.
* **A-FR.3:** Create a reusable `ShortcutIcon` composable that renders the [Esc] and [↵] glyphs next to button labels.

---

## **4. UX & Acceptance Criteria**
1.  **Scenario:** User opens the "Take Over" dialog.
    * *Action:* User presses `Esc`.
    * *Result:* Dialog closes immediately; no data is sent to `/api/chores/{id}/claim`.
2.  **Scenario:** User is typing a new chore title.
    * *Action:* User presses `Enter`.
    * *Result:* The `POST /api/chores/add` request is triggered immediately.
3.  **Scenario:** A confirmation dialog appears.
    * *Observation:* The "Confirm" button has a visible Enter key icon, and the "Cancel" button has a visible Escape key icon.
