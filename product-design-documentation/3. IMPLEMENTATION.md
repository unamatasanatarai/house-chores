# **Implementation: ChoreLoop**

**Status:** Ready \
**Date:** April 24, 2026

## **1. Core System Architecture**
Based on the HLD, the implementation must adhere to a **Single Source of Truth (SSOT)** model.

* **Backend:** PHP RESTful API (Stateless).
* **Android:** MVVM + Jetpack Compose + Room (Cache only) + DataStore (Identity).
* **Web:** ES6 Modules + Vanilla JS + `localStorage` (Identity).
* **State Policy:** Always-online. No offline write support. Conflict resolution follows "First Write Wins" via Server Validation.

---

## **2. Technical Build Sequence**

### **Phase 1: The Foundation (Infrastructure & Identity)**
* **T1: Database Schema Deployment**
    * Implement `users`, `chores`, and `activity_logs` tables as per HLD Section 8.
* **T2: Identity API & Client Persistence**
    * **Backend:** `POST /api/users/add` and `GET /api/users`.
    * **Android:** Implement `IdentityRepository` using Jetpack DataStore.
    * **Web:** Implement `auth.js` module using `localStorage`.
* **T3: API Contract Scaffolding**
    * Set up standardized JSON response wrappers (Success/Error schemas from HLD 7.3).
    * Implement `Bearer [TOKEN]` header validation across all endpoints.

### **Phase 2: The Core Loop (Chore Lifecycle)**
* **T4: CRUD & State Machine**
    * Backend logic for `status` transitions: `available` ↔ `claimed` ↔ `completed` ↔ `archived`.
    * **Guardrails:** Enforce that only the current `claimed_by` user can execute `unclaim` or `done`.
* **T5: Conflict Handling (The 409 Logic)**
    * Implement server-side checks for `PUT /api/chores/{id}/claim`. If `claimed_by` is already set, return `409_CONFLICT_ALREADY_CLAIMED`.
* **T6: Multi-Section Dashboard**
    * Build the 4-section UI: *Available, Claimed by Me, Claimed by Others, Completed (Recent)*.
    * Implement sorting: **Overdue** (calculated client-side) at top, then **Due Date ASC**.

### **Phase 3: Soft Accountability & UX Safety**
* **T7: Activity Logging**
    * Hook into chore actions to populate the `activity_logs` table (Actions: `created`, `claimed`, `unclaimed`, `completed`, `archived`).
    * Build `GET /api/logs?chore_id=UUID`.
* **T8: Safety Mechanisms**
    * **Undo:** Implement the 5-second Snackbar logic on the client that triggers the `unarchive` or `unclaim` endpoint.
    * **Take Over:** Implement the "Take Over" flow with the required confirmation dialog for chores owned by others.

### **Phase 4: Quality Assurance & Validation**
This phase ensures the system's "Flat Trust Model" is enforced correctly by the server and that the UI remains consistent across all clients.

* **State Machine Verification:** * Automated testing of the `PUT /claim` and `PUT /done` endpoints to ensure invalid transitions (e.g., completing an unclaimed chore) are blocked with a `409 Conflict`.
* **Concurrency Stress Testing:** * Simulated "Race Condition" tests where multiple clients attempt to claim the same `chore_id` simultaneously to verify the "First Write Wins" logic.
* **Identity Persistence Checks:** * Validation of `DataStore` (Android) and `localStorage` (Web) to ensure the user's UUID is never lost during app updates or cache clears, preventing "Anonymous" actions.
* **Network Interruption Handling:** * Testing the "Always-Online" requirement by simulating high latency and total network failure during mutation requests to verify the "Offline Overlay" and Error Snackbar triggers.

## **3. Detailed Task Breakdown**

### **Data Models (SQL)**
```sql
-- Chores Table (HLD Section 8)
CREATE TABLE chores (
    id UUID PRIMARY KEY,
    title VARCHAR(255),
    description TEXT,
    status VARCHAR(30),
    due_date TIMESTAMP,
    created_by UUID,
    claimed_by UUID,
    completed_by UUID,
    archived_by UUID,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    claimed_at TIMESTAMP,
    completed_at TIMESTAMP,
    archived_at TIMESTAMP
);
```

### **API Implementation (PHP)**
* **Middleware:** Ensure every request checks for `Authorization: Bearer [TOKEN]`.
* **Validation:** Use `422_VALIDATION_ERROR` for missing title or invalid ISO8601 dates.

### **Android (Compose)**
* **MVVM ViewModels:** `ChoreViewModel` handles polling logic to keep the UI fresh (Refetch on app start/action).
* **UI Components:** `ChoreCard` with conditional action buttons based on `status` and `identity`.

## **4. Risks & Technical Mitigations**

| Risk | HLD Mitigation Strategy | Implementation Action |
| :--- | :--- | :--- |
| **Race Conditions** | First write wins | Server must validate `claimed_by IS NULL` before updating a claim. |
| **Stale UI** | Refetch / Polling | Clients must trigger a `GET /api/chores` immediately following any `PUT` action. |
| **Identity Loss** | Local Storage Persistence | Android: Use `DataStore` for atomic updates. Web: Simple `localStorage` check on every route change. |
| **Connectivity** | Always-online requirement | Interceptor on API calls to show a global "Offline" overlay if status code is unreachable. |

## **5. Milestone Exit Criteria**
1.  **M1 (Identity):** App starts, forces user selection, saves ID, and navigates to (empty) dashboard.
2.  **M2 (Core Loop):** User A creates chore; User B claims it (User A sees it move to "Claimed by Others"); User B completes it.
3.  **M3 (Accountability):** Clicking a chore shows a list of who did what and when; Archiving a chore can be undone within 5 seconds.
