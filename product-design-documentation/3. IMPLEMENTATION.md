## 1. PRD Summary
* **Core Objective:** A zero-hierarchy, high-transparency chore management system for a single household.
* **Target Users:** Household members (Adults/Children) on Web and Android.
* **Key Features:** Identity persistence (local), Chore Lifecycle (Create/Claim/Complete), Soft Accountability (Activity Logs), and Flat Trust (Equal permissions).
* **Technical Complexity:** Low to Moderate. The primary challenge is maintaining state consistency across clients without WebSockets (relying on polling/refetch) and handling race conditions in a "First Write Wins" model.

---

## 2. System Architecture Overview
The system follows a classic **Client-Server Architecture**.

* **Backend (PHP REST API):** Stateless, handling business logic, state transitions, and logging.
* **Database (SQLite):** Relational storage for Chores, Users, and Activity Logs.
* **Android Client (Jetpack Compose):** MVVM architecture, using Retrofit for networking and DataStore for local identity.
* **Web Client (Vanilla JS):** Component-based rendering using the Fetch API and localStorage.



---

## 3. Technical Sequence Roadmap (Ordered Steps)

### **Step 1: Data Schema & API Contract (Sequential)**
Before UI development, the contract must be fixed.
* Define SQL schemas for `users`, `chores`, and `activity_logs`.
* Establish the REST API specification (OpenAPI 3.0).

### **Step 2: Backend Core & Identity (Parallelizable)**
* Build the PHP foundation (routing, DB connection).
* Implement Identity endpoints.
* *Parallel:* Clients (Android/Web) implement the "Identity Picker" and local persistence logic.

### **Step 3: The "Chore Loop" State Machine (Sequential)**
* Backend implements transition logic (e.g., a chore cannot be "Completed" unless "Claimed").
* Implement `409 Conflict` logic for simultaneous claims.

### **Step 4: Cross-Platform UI Development (Parallelizable)**
* Android and Web teams build the Dashboard concurrently, consuming the established API.

## 5. Detailed Task Breakdown

### **M1: Identity Foundation**
* **Backend:**
    * Create `users` table: `id (UUID)`, `name (VARCHAR)`, `created_at`.
    * `POST /api/users/add`: Register name, return UUID.
    * `GET /api/users`: List all members for selection.
* **Frontend (Android/Web):**
    * Implement "First Run" logic: check `DataStore`/`localStorage` for `user_id`.
    * View: User selection/creation screen.

### **M2: The Core Loop (MVP)**
* **Database:**
    * Create `chores` table: `id`, `title`, `description`, `status`, `created_by`, `claimed_by`, completed_by, `archived_by`, `due_date`, `created_at`, `claimed_at`, `completed_at`, `archived_at`.

* **Backend:**
    * `GET /api/chores`: Support filtering by status and `assigned_to`.
    * `PUT /api/chores/{id}/claim`: Logic—check if `claimed_by` is null; if not, return `409`.
    * `PUT /api/chores/{id}/done`: Logic—verify current user is the owner.
* **Frontend (Android/Web):**
    * Dashboard: 4-column/section layout.
    * Overdue Logic: Client-side comparison of `due_date` vs `NOW()`.

### **M3: Soft Accountability**
* **Database:**
    * Create `activity_logs`: `id`, `chore_id`, `user_id`, `action`, `timestamp`.
* **Backend:**
    * Trigger log insertion on every chore state change.
    * `PUT /api/chores/{id}/unarchive`: Restore chore to "Available" or "Completed."
* **Frontend:**
    * Chore Detail View: Fetch and render log list.
    * Snackbar Service: Trigger on "Archive" action with `unarchive` API callback.

## 6. Risks & Mitigations

| Risk | Impact | Mitigation Strategy |
| :--- | :--- | :--- |
| **Stale Data (No Push)** | High | Implement "Smart Refetching": refresh data on window focus (Web) or `onResume` (Android), and after every local action. |
| **Race Conditions** | Medium | Server-side validation is mandatory. Clients must handle `409 Conflict` errors by triggering a mandatory data refresh and showing a "Too Late!" message. |
| **Flat Model Abuse** | Low | This is a social risk, not technical. Mitigation is the **Activity Log**—making "trolling" visible to the whole house. |
| **Security** | Medium | Since there is no password, use the UUID as a `Bearer Token`. It’s not banking-grade, but sufficient for a household "trust" model. |

---

## 7. Assumptions
1.  **Network:** All devices are on the same network or have access to a public IP where the PHP backend is hosted.
2.  **Environment:** The PHP environment supports basic PDO for secure database interactions.
3.  **Concurrency:** The volume of chores per household is low enough (<500 active) that simple polling will not degrade performance.
