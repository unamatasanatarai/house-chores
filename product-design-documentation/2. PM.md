### **1. User Stories**
Following our **Flat Trust Model**, these stories treat every user as an equal participant in the household ecosystem.

#### **Theme: Identity & Presence**
* **Story 1.1:** As a family member, I want to create a Family Name on first launch so we have a shared digital space without needing an account.
* **Story 1.2:** As a family member, I want to add my name to the family list so I can start participating in chores.
* **Story 1.3:** As a family member, I want to select my name from the user list and have the app remember me so I don’t have to pick my profile every time I open the app.
* **Story 1.4:** As a family member, I want to see my name in the top navigation bar at all times so I am confident I am acting as myself and not a sibling.

#### **Theme: Chore Lifecycle (The Core Loop)**
* **Story 2.1:** As a family member, I want to create a chore with a title and due date so others know what needs to be done.
* **Story 2.2:** As a family member, I want to see a list of "Available" chores so I can choose a task to take on.
* **Story 2.3:** As a family member, I want to claim a chore so that it is locked to me and others know I am responsible for it.
* **Story 2.4:** As a family member, I want to mark my claimed chore as "Done" so I can feel a sense of accomplishment and clear my list.
* **Story 2.5:** As a family member, I want to delete any chore, but only after confirming a "Are you sure?" dialog, to prevent accidental data loss.

#### **Theme: Data & Connectivity**
* **Story 3.1:** As a family member, I want the app to require an active connection to make changes, ensuring that I always see the most up-to-date state of chores.
* **Story 3.2:** As a family member, I want deleted chores to be soft-deleted so that we can recover data if a mistake is made.

### **2. Prioritization (MoSCoW Method)**
To ensure we hit our MVP timeline, we must be ruthless with our "Must-Haves."

| Category | Features / Stories | Rationale |
| :--- | :--- | :--- |
| **Must Have** | Identity setup (1.1, 1.2, 1.3), Name in Top Bar (1.4), The "Claim-Complete" loop (2.1-2.4), Confirm Deletion (2.5), PHP/MySQL API with Hardcoded Token. | Without these, the app has no functional utility or identity. These are the non-negotiables. |
| **Should Have** | Soft-delete recovery endpoints and advanced error handling for network drops. | The app is strictly online-only, so robust API error handling and data safety are necessary for a smooth experience. |
| **Could Have** | Local notifications for overdue tasks, visual confetti on completion, chore descriptions. | These add "delight" and engagement but are not required for the core utility of tracking a chore. |
| **Won't Have** | Avatars, Email/Password auth, Parent/Child roles, Photo proof. | These are explicitly excluded from the current design to maintain the "Flat Trust" and "No-Auth" speed. |

### **3. Release Planning**
We will follow a phased release to mitigate the technical risk of the synchronization logic.

#### **V1.0: The "Connected" MVP**
* **Objective:** Validate the core loop and backend integration.
* **Scope:** * Full identity setup and profile selection.
    * Real-time CRUD for chores (requires internet).
    * PHP/MySQL API with Hardcoded Token authentication.
    * Confirmation dialogs and Top Bar identity display.
* **Goal:** A functional app where a family can coordinate chores if they have a Wi-Fi connection.

#### **V1.1: The "Safe & Recoverable" Update**
* **Objective:** Enhance data safety and handle edge cases gracefully.
* **Scope:**
    * UI for viewing/recovering soft-deleted chores (using the `deleted_at` field).
    * Enhanced network error handling to inform users when they are offline and prevent them from initiating actions.
* **Goal:** Ensuring the online-only experience is robust and accidental data loss is completely mitigated.

#### **V1.2: The "Habit" Update**
* **Objective:** Drive long-term engagement.
* **Scope:**
    * Local notifications (8:00 AM due date/overdue).
    * Visual "Success" states (Confetti, micro-copy).
    * Chore descriptions.
