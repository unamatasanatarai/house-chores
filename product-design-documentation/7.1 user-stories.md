# 1. Overview of Features/Epics

* **Epic: Keyboard Interaction Engine**
    Manages the global `keydown` listener and the LIFO (Last-In-First-Out) **Modal Stack** specifically for Dialogs and Forms.
* **Epic: Contextual Navigation & Confirmation**
    Implementation of `Esc` for dismissal and `Enter` for submission, with strict exclusions for textareas and specific non-cancellable overlays.
* **Epic: Shortcut Affordance & Visual Feedback**
    Visual "key-cap" hints for buttons, specifically within modal/dialog contexts.

---

## 2. User Stories

### Epic: Contextual Navigation & Confirmation

#### **Title: Global Escape Key Dismissal**
* **User Story:** *As a user, I want to press the `Esc` key to dismiss active dialogs, so that I can quickly return to the main interface.*
* **Description:** Implement a global listener to intercept `Esc`. This listener only targets the top-most **Modal or Dialog**.
* **Acceptance Criteria:**
    * Pressing `Esc` closes the top-most Confirmation Dialog/Modal.
    * **Exclusion:** `Esc` performs no action when the Identity Selection overlay is active.
    * **Exclusion:** `Esc` does **not** affect or dismiss Snackbars.
    * Closing via `Esc` triggers no API actions.
* **Priority:** High

#### **Title: Contextual Enter Key Confirmation**
* **User Story:** *As a user, I want to press the `Enter` key to confirm actions, so that I can complete tasks without shifting focus to the mouse.*
* **Description:** Trigger the "Positive" action in a modal or form via `Enter`, provided validation is met and the user is not in a multi-line input.
* **Acceptance Criteria:**
    * `Enter` triggers the "Confirm/Submit" button action if enabled.
    * **Exclusion:** If the focus is within a `<textarea>`, the `Enter` key must insert a newline as per standard behavior and **not** trigger a submission.
    * **Exclusion:** `Enter` does **not** dismiss or interact with Snackbars.
* **Priority:** High

---

### Epic: Shortcut Affordance & Visual Feedback

#### **Title: Visual Key-cap Shortcut Hints**
* **User Story:** *As a user, I want to see shortcut icons on modal buttons, so that I know which keys will trigger the action.*
* **Description:** Update templates to display `[鈫礭` and `[Esc]` icons.
* **Acceptance Criteria:**
    * Buttons within **Dialogs and Form-based Modals** display the respective key-cap icons.
    * **Constraint:** `ChoreCard` components only display these icons when rendered within a Modal context; they do not appear on standard board/list view cards.
* **Priority:** Medium

## 3. Documentation & Technical Logic

| Component | `Esc` Behavior | `Enter` Behavior | Shortcut Icons |
| :--- | :--- | :--- | :--- |
| **Modals / Dialogs** | Dismisses top-most | Confirms (if enabled) | Yes (`[Esc]` / `[鈫礭`) |
| **Textareas** | Dismisses Modal | **New Line (Standard)** | No |
| **Snackbars** | **No Action** | **No Action** | No |
| **Identity Selection**| **No Action** | No Action | No |
| **ChoreCard (List)** | No Action | No Action | **No** |

## 4. Potential Risks or Gaps

* **User Expectations (Snackbars):** Since Snackbars are no longer dismissible via keyboard, users who are accustomed to "clearing the screen" with `Esc` might find the persistent notifications frustrating. We should ensure the Snackbar auto-dismiss timer is well-tuned.
* **Focus Trapping:** When a Modal is open, we must ensure focus doesn't "leak" to elements behind it in the DOM, otherwise `Enter` might trigger actions on the background page instead of the Modal.
* **Template Logic:** The `ChoreCard` template needs a conditional check (e.g., `isModalView`) to ensure shortcut icons are only injected when appropriate.
