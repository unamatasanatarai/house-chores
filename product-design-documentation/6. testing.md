
# **Testing Strategy: ChoreLoop**

**Status:** Implementation-Ready \
**Focus:** State Machine Integrity & Concurrency Validation

## **1. Testing Philosophy**
ChoreLoop relies on a **Flat Trust Model** and **Always-Online** connectivity. Testing is prioritized around the **Chore Lifecycle** to ensure that state drift never occurs between the server and the multi-platform clients.

---

## **2. Test Levels & Tooling**

### **2.1 Backend Unit & Integration (PHPUnit)**
* **State Machine Validation:** Ensure every status transition (Available → Claimed → Completed) follows the logic defined in the HLD.
    * *Edge Case:* Attempting a `PUT /done` on a chore that is still `available` must return a `409_CONFLICT_INVALID_STATE`.
* **Identity Middleware:** Verify that requests without a valid `Bearer [TOKEN]` are rejected with `401_UNAUTHORIZED`.

### **2.2 Android UI & Logic (JUnit / Espresso)**
* **ViewModel Polling:** Test that the `ChoreViewModel` correctly triggers a refetch after a successful `PUT` action.
* **DataStore Persistence:** Verify that the user’s identity survives a process death/restart.

### **2.3 Web Component Testing (Vitest / Cypress)**
* **Undo Logic:** Verify the 5-second window for the "Undo" Snackbar.
    * *Test:* Archive a chore → Click Undo → Confirm status reverts via API call.

---

## **3. Critical "High-Signal" Test Cases**

| Test Case ID | Scenario | Expected Outcome | Signal |
| :--- | :--- | :--- | :--- |
| **TC-01** | **Race Condition: Simultaneous Claim** | Server receives two identical `/claim` requests at the same millisecond; rejects second request with `409`. | **Concurrency Handling** |
| **TC-02** | **Unauthorized Ownership Takeover** | User B attempts to `unclaim` a chore currently owned by User A. Server returns `403_FORBIDDEN`. | **System Security** |
| **TC-03** | **Identity Loss Recovery** | Clear `localStorage`/`DataStore`; App must force re-entry to the Identity Selection screen. | **Persistence Logic** |
| **TC-04** | **Polling Sync** | User A completes a chore on Web; User B’s Android dashboard updates state to "Completed" on next poll. | **SSOT Integrity** |

---

## **4. Performance & Reliability Testing**

* **Connectivity Stress Test:** Using network throttling (Chrome DevTools / Android Emulator), verify the "Offline Overlay" appears when the API is unreachable.
* **Activity Log Load:** Ensure `GET /api/logs` remains performant (under 200ms) even as the `activity_logs` table grows to 10k+ rows.
