### **Technical Project Management (TPM) Document: Family Chores App**

This document outlines the final technical specifications, database schema, and integration strategy for the Family Chores App, encompassing both Android and Web implementations.

---

### **1. MySQL Schema**
To support the "Flat Trust" model, we use a normalized **InnoDB** schema to ensure data integrity through foreign keys.

#### **Table: `families`**
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | VARCHAR(36) | PK | UUID. |
| `family_name` | VARCHAR(100) | NOT NULL | Shared household name. |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation metadata. |

#### **Table: `users`**
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | VARCHAR(36) | PK | UUID. |
| `family_id` | VARCHAR(36) | FK -> families(id) | Scopes user to a family. |
| `name` | VARCHAR(50) | NOT NULL | User's display name. |
| `updated_at` | TIMESTAMP | ON UPDATE CURRENT_TIMESTAMP | Tracking metadata. |

#### **Table: `chores`**
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | VARCHAR(36) | PK | UUID. |
| `family_id` | VARCHAR(36) | FK -> families(id) | Scopes chore to the family. |
| `title` | VARCHAR(255) | NOT NULL | Task name. |
| `description` | TEXT | NULL | Optional task details. |
| `status` | VARCHAR(20) | NOT NULL | 'available', 'claimed', 'completed'. |
| `assigned_to` | VARCHAR(36) | NULL, FK -> users(id) | The claimant. |
| `due_date` | DATETIME | NOT NULL | Deadline for tasks. |
| `deleted_at` | DATETIME | NULL | Soft delete timestamp. |

### **2. API Contract & Security**
Every request must include the header `X-CHORES-TOKEN: [HARDCODED_STRING]`. The backend must also support CORS to enable Webapp functionality.

* **CORS Headers (PHP):** The server must allow requests from browser-based clients by setting `Access-Control-Allow-Origin: *` and supporting `X-CHORES-TOKEN` in `Access-Control-Allow-Headers`.
* **Endpoint: `GET /chores.php`**: Returns active chores where `deleted_at IS NULL`.
* **Endpoint: `POST /action.php`**: Handles `claim`, `complete`, and `delete` actions.

---

### **3. Synchronization & Identity Strategy**
We utilize a **"First to the Server Wins"** model where the server is the Single Source of Truth (SSOT).

#### **Conflict Resolution**
* If two users claim the same chore, the first request accepted by the server sets `assigned_to`.
* Subsequent requests receive a **409 Conflict** error.
* Clients (Android/Web) must handle this by showing a "Someone claimed this just before you" message.

#### **Identity Persistence**
* **Android:** Uses **Jetpack DataStore** to persist `activeUserId` locally.
* **Webapp:** Uses **`localStorage`** to store `activeUserId` across browser sessions.

### **4. Environment Orchestration**

| Component | Technology Stack | Role |
| :--- | :--- | :--- |
| **Backend** | Vanilla PHP 8.x + MySQL | REST API and SSOT database. |
| **Android** | Kotlin, Compose, MVVM | Native mobile client with DataStore. |
| **Webapp** | Pure HTML, CSS, Vanilla JS | Zero-build, browser-based client using Fetch API. |

---

### **5. Key Client Architecture**

#### **Android (Clean Architecture)**
* **Data:** Retrofit service and DataStore implementation.
* **Domain:** Use cases for `ClaimChore` and `SyncChore`.
* **UI:** Compose widgets and ViewModels for the Dashboard and Profile.

#### **Webapp (Vanilla SPA)**
* **`index.html`**: Container for the dashboard.
* **`app.js`**: Manages state, handles `fetch()` calls, and updates the DOM dynamically based on API responses.
* **`navigator.onLine`**: Monitoring tool to disable UI interactions during network loss.
