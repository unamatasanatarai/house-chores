## **1. Backend & RESTful API Tasks (PHP)**

### **Phase 1: Infrastructure & Data**
* **T1.1: Database Schema Migration** – Create the `users`, `chores`, and `activity_logs` tables in MySQL as defined in the Data Model.
* **T1.2: Base API Controller & Routing** – Set up the entry point for the REST API and the routing logic for `/api/*` endpoints.
* **T1.3: Response Wrapper Implementation** – Build a standardized utility to ensure all responses follow the Success/Error JSON schemas (e.g., `success`, `data`, `meta`, `error` blocks).
* **T1.4: Middleware: Identity Validation** – Implement a check for the `Authorization: Bearer [TOKEN]` header to validate that a `user_id` exists before allowing any requests.

### **Phase 2: Identity & User Endpoints**
* **T2.1: POST /api/users/add** – Create the endpoint to register a new user name and return a unique UUID.
* **T2.2: GET /api/users** – Create the endpoint to fetch the list of existing users for the initial selection screen.

### **Phase 3: Chore Lifecycle & Business Logic**
* **T3.1: GET /api/chores** – Implement fetching chores with support for status filtering and the sorting logic (Overdue → Due Date ASC).
* **T3.2: POST /api/chores/add** – Create the endpoint to add a new chore, validating the title and ISO8601 date format.
* **T3.3: PUT /api/chores/{id}/claim** – Implement the claim logic with a "First Write Wins" check: return a `409_CONFLICT_ALREADY_CLAIMED` if the chore is already taken.
* **T3.4: PUT /api/chores/{id}/done** – Implement completion logic: ensure only the user who claimed the chore can mark it as done.
* **T3.5: PUT /api/chores/{id}/archive** – Create the endpoint to move a chore to the archived state.
* **T3.6: PUT /api/chores/{id}/unclaim & /unarchive** – Implement the endpoints needed to support the "Undo" safety mechanism.

### **Phase 4: Accountability Logging**
* **T4.1: Activity Log Observer** – Create a backend hook that automatically inserts a row into `activity_logs` whenever a chore status changes.
* **T4.2: GET /api/logs** – Create the endpoint to retrieve the history for a specific chore ID.

## **2. Web App Tasks (Vanilla JS)**

### **Phase 1: Core Architecture**
* **W1.1: Project Scaffolding** – Set up the HTML5/CSS grid structure and ES6 module entry points.
* **W1.2: API Service Module** – Create a `fetch` wrapper that automatically attaches the stored `Bearer [TOKEN]` to every request.
* **W1.3: Central State Store** – Implement a basic state management object to hold the current list of chores and the active user’s identity.

### **Phase 2: Identity Selection (The "Gate")**
* **W2.1: Identity Selection UI** – Build the modal/screen that fetches users from `GET /api/users` and allows a selection or name entry.
* **W2.2: LocalStorage Persistence** – Implement the logic to save the selected UUID to `localStorage` and skip the selection screen on subsequent visits.

### **Phase 3: Dashboard & Components**
* **W3.1: Multi-Section Dashboard Layout** – Build the 4-section grid (Available, Claimed by Me, Claimed by Others, Completed).
* **W3.2: Chore Card Component** – Create a reusable JS component to render chore details, including conditional buttons (e.g., "Take Over" shows if someone else owns it).
* **W3.3: Refetch/Polling Logic** – Implement the "Always-Online" requirement by triggering a full chore list refetch after every successful action.

### **Phase 4: Safety & UX**
* **W4.1: Undo Snackbar** – Implement the 5-second snackbar notification that appears after archiving/unclaiming, providing a button to call the "Undo" endpoints.
* **W4.2: Confirmation Dialogs** – Build the modal for the "Take Over" flow to prevent accidental reassignments.
* **W4.3: Connection Interceptor** – Add a global listener to show an "Offline" overlay if the API becomes unreachable.

## **3. Testing & QA Tasks**

### **Backend Validation (PHPUnit)**
* **T5.1: Unit Test: Chore State Transitions** – Write tests for the `Chore` model to ensure it rejects invalid status updates.
* **T5.2: Integration Test: Middleware Auth** – Verify that any request without a valid `Bearer` token returns a `401 Unauthorized`.
* **T5.3: Concurrency Test: Claim Collision** – Script a test that fires concurrent requests to the same ID to verify only one `200 OK` is returned while others receive `409 Conflict`.

### **Android & Web Frontend Testing**
* **W5.1: UI Test: The "Undo" Window** – Verify that the Archive action triggers a 5-second timer and that clicking "Undo" successfully calls the `PUT /unarchive` endpoint.
* **W5.2: Integration Test: Dashboard Polling** – Ensure that a successful action (like "Claim") triggers a background refresh of the chore list.
* **W5.3: E2E: Onboarding Flow** – Automate a test that starts with a "Clean State," selects a user, and verifies navigation to the Dashboard.

### **Manual QA & Edge Cases**
* **M5.1: Cross-Platform Sync Test** – Manually verify that marking a chore as "Done" on the Web client reflects on the Android client within one polling cycle.
* **M5.2: "Take Over" UX Verification** – Verify the confirmation modal triggers correctly when attempting to claim a chore already owned by another user.
* **M5.3: Connectivity Interceptor** – Test the global "Offline" banner by toggling Airplane Mode during active app use.
