## 1. Overview of Epics
* **Epic 1: Identity & Persistence:** Establishing who the user is and ensuring the device "remembers" them.
* **Epic 2: Chore Lifecycle Management:** The core CRUD and state transition engine (Create, Claim, Complete).
* **Epic 3: Dashboard & Visibility:** Organizing tasks by priority and ownership.
* **Epic 4: Soft Accountability & History:** Tracking "who did what" to maintain trust without rigid permissions.
* **Epic 5: Safety & Reliability:** Error handling, connectivity, and accidental action recovery.

---

## 2. User Stories

### **Epic 1: Identity & Persistence**

#### **Story 1.1: Initial Identity Selection**
* **User Story:** As a new user, I want to select or create my name upon first launch, so that my actions are correctly attributed to me.
* **Description:** A forced "onboarding" screen for users to establish their identity before accessing the dashboard.
* **Acceptance Criteria:**
    * App displays a list of existing users or an option to add a new name.
    * Once selected, the UUID is stored in `localStorage` (Web) or `DataStore` (Android).
    * The user is redirected to the Dashboard and cannot return to the selection screen via the "Back" button.
* **Priority:** High
* **Dependencies:** Backend T2 (Identity API).

#### **Story 1.2: Persistent Identity Visibility**
* **User Story:** As a user, I want to see my name in the app header at all times, so I know which profile I am currently using.
* **Description:** A persistent UI indicator of the active session.
* **Acceptance Criteria:**
    * TopAppBar displays "Acting as: [User Name]".
    * If local storage is cleared, the app must force a re-selection.
* **Priority:** Medium
* **Dependencies:** Story 1.1.

---

### **Epic 2: Chore Lifecycle Management**

#### **Story 2.1: Create a New Chore**
* **User Story:** As a household member, I want to create a chore with a title and due date, so that others know what needs to be done.
* **Description:** Adding a new task to the shared household list.
* **Acceptance Criteria:**
    * User can input a Title (Required) and Description/Due Date (Optional).
    * System automatically assigns `created_by` and `created_at` fields.
    * The chore status defaults to `available`.
* **Priority:** High
* **Dependencies:** Backend T1 (DB Schema).

#### **Story 2.2: Claim an Available Chore**
* **User Story:** As a contributor, I want to claim an available chore, so that the household knows I am taking responsibility for it.
* **Description:** Moving a chore from "Available" to "Claimed" status.
* **Acceptance Criteria:**
    * User can click "Claim" on any `available` chore.
    * Chore status updates to `claimed` and `claimed_by` is set to the current user.
    * The chore moves to the "Claimed by Me" section for the actor.
* **Priority:** High
* **Dependencies:** Story 2.1.

#### **Story 2.3: Mark Chore as Complete**
* **User Story:** As the chore owner, I want to mark my claimed chore as complete, so that the household knows the task is finished.
* **Description:** Closing the loop on a claimed task.
* **Acceptance Criteria:**
    * Only the user in the `claimed_by` field can click "Complete."
    * Sets `completed_at` and `completed_by` timestamps.
    * Chore moves to the "Completed (Recent)" section.
* **Priority:** High
* **Dependencies:** Story 2.2.

---

### **Epic 3: Dashboard & Visibility**

#### **Story 3.1: Intelligent Task Sorting**
* **User Story:** As a user, I want chores to be sorted by urgency, so I can prioritize the most important tasks.
* **Description:** Implementing the sorting logic defined in HLD 4.3.
* **Acceptance Criteria:**
    * Overdue chores (Claimed or Available) appear at the very top.
    * Chores are sorted by `due_date` in ascending order (ASC).
    * Dashboard is split into 4 sections: Available, Claimed by Me, Claimed by Others, Completed.
* **Priority:** Medium
* **Dependencies:** Story 2.1.

---

### **Epic 4: Soft Accountability**

#### **Story 4.1: Take Over a Chore**
* **User Story:** As a user, I want to take over a chore claimed by someone else, so I can finish it if they are unable to.
* **Description:** A "Flat Trust" mechanism to reassign ownership.
* **Acceptance Criteria:**
    * Clicking "Take Over" on a chore claimed by another user triggers a confirmation dialog.
    * Upon confirmation, `claimed_by` is updated to the current user.
    * The action is logged in the Activity Log.
* **Priority:** Medium
* **Dependencies:** Story 2.2.

#### **Story 4.2: View Chore Activity Log**
* **User Story:** As a user, I want to see the history of a chore, so I can understand its progress and who has interacted with it.
* **Description:** Visibility into the lifecycle events of a task.
* **Acceptance Criteria:**
    * Chore detail view shows at least the last 10 actions (created, claimed, etc.).
    * Each entry shows the User Name and a relative timestamp (e.g., "2 hours ago").
* **Priority:** High
* **Dependencies:** T7 (Activity Logging API).

## 3. Non-Functional User Stories

#### **Story 5.1: Real-time Consistency (Polling)**
* **User Story:** As a user, I want my dashboard to stay updated with others' actions, so I don't try to claim a chore someone else just took.
* **Description:** Maintaining a Single Source of Truth (SSOT).
* **Acceptance Criteria:**
    * The app triggers a refetch (`GET /api/chores`) immediately after any local action.
    * (Optional but recommended) Periodic polling ensures the UI is fresh even without user input.
* **Priority:** High

#### **Story 5.2: Graceful Conflict Resolution**
* **User Story:** As a user, I want to be notified if my action fails due to a conflict, so I can see the updated state of the app.
* **Description:** Handling "First Write Wins" server logic.
* **Acceptance Criteria:**
    * If a user tries to claim a chore already claimed by another, the app displays a `409 Conflict` error message.
    * The UI automatically refreshes to show the current server state.
* **Priority:** High

## 4. Assumptions & Clarifications Needed
1.  **"Completed (Recent)" Logic:** The PRD mentions keeping completed chores visible, but the HLD mentions an "Archive" state. 
    * *Assumption:* Completed chores stay in the "Completed" section until a user manually archives them, or after a certain "recency" window (TBD).
2.  **Authentication:** The documents mention `Authorization: Bearer [TOKEN]` but also say "No authentication (email/social)." 
    * *Assumption:* The "Token" is a simple UUID generated during the "Identity Selection" phase, not a secure JWT from a login service.
3.  **Token Exposure:** It is explicitly accepted as an MVP risk that tokens are not encrypted/secured.

---

## 5. Potential Risks or Gaps
* **The "Nag" Factor:** Without push notifications (Out-of-Scope), the "Claim Velocity" (Success Metric) depends entirely on users habituating themselves to opening the app.
* **Trolling/Accidental Deletion:** Because any user can "Archive" someone else's chore, there is a risk of data loss. 
    * *Mitigation:* The 5-second "Undo" snackbar and Activity Log are the only defenses.
* **Network Dependency:** Since there is no offline support, the app is unusable in "dead zones" of a house (e.g., basement/garage), which is a common environment for chores.
