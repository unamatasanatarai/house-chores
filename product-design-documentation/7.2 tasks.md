## 1. Context & Objective
The goal is to implement a standardized keyboard interaction system using a **LIFO (Last-In-First-Out) stack** to manage UI dismissals and confirmations via `Esc` and `Enter`. 

* **Logic:** Centralized in `app.js`.
* **UI:** Distributed across component templates with universal "key-cap" styling.

---

## 2. Task Breakdown

### **T1: LIFO Modal Stack Utility**
* **Description:** Create a simple array-based manager in the global state to track active, dismissible UI elements.
* **Acceptance Criteria:**
    * Expose `push(element)` and `pop()` methods.
    * **KISS:** Only Modals/Dialogs are added. Identity Selection and Snackbars are simply ignored (not pushed).
* **Priority:** High

### **T2: Global Keydown Listener & Escape Logic**
* **Description:** Add a window-level listener to intercept the `Escape` key.
* **Acceptance Criteria:**
    * Intercepts `Esc`.
    * If the stack is not empty, calls the close method on the top-most element.
    * Ensures focus is returned to the element that triggered the modal.
* **Priority:** High

### **T3: Contextual Enter Confirmation**
* **Description:** Implement `Enter` key confirmation within the global listener.
* **Acceptance Criteria:**
    * Triggers the "Positive" action button if it exists and is enabled.
    * **Exclusion:** If the user is in a `<textarea>`, `Enter` must perform a standard newline.
    * Prevents default form submission/refresh.
* **Priority:** High

### **T4: Key-cap CSS & Icon Assets**
* **Description:** Implement the visual styling for the shortcut hints.
* **Acceptance Criteria:**
    * CSS `.key-cap`: Monospace font, light gray background, subtle border.
    * Provide inline SVGs for `[↵]` and `[Esc]`.
    * **Note:** These remain visible on all devices, including mobile/touch.
* **Priority:** Medium

### **T5: Template Integration (Dialogs & ChoreCards)**
* **Description:** Update HTML templates to display the new icons.
* **Acceptance Criteria:**
    * Inject `[↵]` on confirm buttons and `[Esc]` on cancel buttons.
    * Update `ChoreCard` to accept an `isModalView` flag; icons only show when this is true.
* **Priority:** Medium

---

## 3. Technical Considerations & Sequencing

### **Sequencing**
1.  **Foundation:** T1 (Stack) -> T2 (Esc) -> T3 (Enter).
2.  **Visuals:** T4 (CSS) can be done in parallel.
3.  **Finalize:** T5 (Templates) integrates the logic and styling.

### **Behavior Matrix**

| Element | Esc Key | Enter Key | Icon Visible |
| :--- | :--- | :--- | :--- |
| **Modal/Dialog** | Close | Submit | Yes |
| **Textarea** | Close | **New Line** | No |
| **Identity Overlay** | No Action | No Action | No |
| **Snackbar** | No Action | No Action | No |

---

## 4. Finalized Strategy
* **No specific blocking:** We simply don't push the Identity Selection overlay to the stack, so the `Esc` listener will find an empty stack and do nothing.
* **Single-action focus:** Since there is only ever one positive action per context, the `Enter` logic will simply target the primary button of the active modal.
* **Platform agnostic:** Key-cap icons will be rendered regardless of the input method (mouse/touch/keyboard) for UI consistency.
